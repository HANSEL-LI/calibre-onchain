# W1.2 — A-lite EIP-712 voucher extension (#422)

Extends the merged W1.1 settlement core (`contracts/src/CalibreMarket.sol`) with a
backend-signed EIP-712 price-voucher `buy` path, under custody Model A-lite. Price
discovery stays in the off-chain LMSR; the contract verifies a voucher signed by
calibre's standing-counterparty key and atomically settles USDC-in + shares-out.

This is a clean, **isolated** extension of the W1.1 core: it does NOT touch
`mint` / `resolve` / `redeem` / `transferShares`, only adds new state, errors,
events, and the `buy(quote, sig)` entrypoint plus a counterparty-inventory primitive.
Isolation is deliberate — if the Saturday-noon W1.3 checkpoint fails and custody
degrades to Model B, this surface can be excised without disturbing the core
(briefing §3).

## The standing-counterparty model (key decision)

The issue's `buy(quote, sig)` makes **calibre the standing counterparty**: the user
pays USDC and receives shares, rather than minting a complete set. To preserve W1.1's
solvent-by-construction invariant (every outstanding YES+NO pair backed by exactly one
USDC unit locked at mint), calibre seeds **inventory** by minting complete sets to a
dedicated `counterparty` address it controls. `buy` then:

1. verifies the EIP-712 voucher signature against `voucherSigner`,
2. enforces expiry / per-user monotonic nonce / `maxCost` (the user's signed slippage
   bound) / optional per-market notional cap,
3. pulls `cost ≤ maxCost` USDC from the buyer **to the counterparty** (the
   counterparty fronted the USDC at mint to lock the set; the buyer reimburses it for
   the side they take), and
4. transfers `size` shares of `side` from `counterparty` inventory to the buyer.

Because shares move from already-minted, already-backed inventory, the contract's USDC
backing for outstanding winning shares is **unchanged by a buy** — the solvency
invariant W1.1 established still holds. The counterparty keeps the complementary
(unsold) side as its position; at resolve it redeems whatever it still holds. No
on-chain pricing: `cost` is whatever the user signed up to (≤ maxCost), set off-chain
by the LMSR.

`voucherSigner` and `counterparty` are distinct roles (signing key vs. inventory
holder) but default-settable to the resolver; both are resolver-managed.

## EIP-712 surface (the W1.2 ↔ W3.1 interface — agreed here, frozen)

- **Domain:** `name="CalibreMarket"`, `version="1"`, `chainId=5042002` (Arc testnet,
  W8 §3), `verifyingContract=address(this)`. Domain separator cached at construction
  and recomputed if `block.chainid` ever differs (fork-safe).
- **Typed struct** (exact field order — W3.1 must match byte-for-byte):
  ```
  Quote(uint256 marketId,address buyer,uint8 side,uint256 size,uint256 maxCost,uint256 nonce,uint256 expiry)
  ```
  `side`: 0 = NO, 1 = YES (matches the `bool isYes` core convention: YES = 1/true).
  `buyer` is bound so a voucher is only usable by its intended user — `msg.sender`
  must equal `q.buyer`. `maxCost` and the pulled `cost` are **6-decimal ERC-20 USDC
  base units** (W8 §3 — never the 18-dec native asset). `nonce` is per-user monotonic.
  `expiry` is a unix timestamp; the off-chain signer sets it ≤ 30s out (W8 §5) — the
  contract only enforces `block.timestamp < expiry`, it cannot police how far out the
  signer set it, so the 30s bound is a signer-side discipline, documented here.

## Bound values (from W8 §5, enforced where they live)

| Bound | Value | Enforced |
|---|---|---|
| `expiry` | ≤ 30s out | off-chain signer (W3.1); contract enforces `< expiry` |
| `maxCost` | LMSR quote × 1.01 | off-chain signer; contract enforces `cost ≤ maxCost` |
| per-market notional cap | optional, ~1,000 USDC | contract-side `marketNotionalCap` (0 = disabled) |
| nonce | per-user monotonic | contract-side `nonces[buyer]` |

The `cost` the contract pulls is itself a signed field? No — `cost` is **not** in the
struct; the contract pulls exactly `maxCost`? That would remove the user's headroom.
Decision: the contract pulls `cost` supplied as a `buy` argument, requires
`cost <= maxCost`, and the off-chain signer guarantees the LMSR price ≤ cost. Simpler
and matches the issue skeleton ("charged USDC ≤ maxCost"). The voucher caps; the
actual charge is the (≤ cap) amount the relayer/user passes. See Decisions.

## Decisions (rationale)

- **Standing-counterparty = pre-minted inventory** (not on-the-fly mint in `buy`):
  keeps the W1.1 solvency proof intact and makes `buy` a pure share-transfer +
  USDC-pull. Minting inside `buy` would change USDC backing math and entangle the
  excisable surface with the core.
- **`cost` as a `buy` argument bounded by signed `maxCost`** (not pulling `maxCost`
  itself): matches the issue skeleton's "charged USDC ≤ maxCost" and preserves the
  user's signed slippage *ceiling* while letting the actual fill be the cheaper LMSR
  price. The signer guarantees `cost` reflects the real quote; `maxCost = quote×1.01`
  is the leak ceiling.
- **`buyer` bound into the struct**: a leaked voucher can't be replayed by a different
  address even before the nonce burns. Belt-and-suspenders with the nonce.
- **`side` as uint8 0/1** (not bool) in the typed struct: EIP-712 has no bool-vs-uint8
  ambiguity, and uint8 is the conventional ABI type W3.1's signer will encode; maps to
  the core's `isYes` at the boundary (`side == 1`).
- **Distinct `voucherSigner` vs `counterparty` roles**: the signing key (hot, in
  backend memory, W8 §5) is a different trust surface from the inventory-holding
  address; separating them lets the signer key rotate without moving inventory.
- **Sibling `CalibreVoucher.t.sol`**: keeps W1.1's test file pristine, reinforcing the
  excise-cleanly property for the W1.3 checkpoint.

## Risks

- **Inventory exhaustion:** a buy beyond seeded inventory reverts on `InsufficientShares`
  in the transfer — acceptable (off-chain signer won't quote beyond inventory); tested.
- **Domain/chainId drift on a fork:** cached separator recomputed when `block.chainid`
  changes — standard OZ-style guard.
- **18-vs-6-dec USDC:** all amounts are 6-dec ERC-20 base units, same as W1.1; no
  native-asset math introduced.
