# W1.1 — custody-independent mint/resolve/redeem core (`CalibreMarket.sol`)

Issue: HANSEL-LI/Calibre#421. Target repo: `calibre-onchain` (this repo).
Umbrella: HANSEL-LI/Calibre#415. Branch: `w1.1-settlement-core`.

## Goal

Replace the W0 scaffold `CalibreMarket.sol` with the **custody-independent
settlement core**: a binary YES/NO market registry where 1 USDC mints a complete
set (1 YES + 1 NO), shares transfer, a resolver settles a market once, and
winners redeem 1:1 for USDC. Solvent by construction — every outstanding YES+NO
pair is backed by exactly 1 USDC locked at mint.

**Out of scope (do not implement here):** EIP-712 voucher buy/redeem signing
(W1.2 #422), the Python SDK (W4.1), any on-chain pricing/LMSR. Leave clean
extension hooks for W1.2 but no voucher logic.

## Load-bearing constraints (from W8 spikes §3)

- **USDC decimals.** Arc's *native* USDC gas token is 18-decimal, but the
  **ERC-20 USDC interface is 6-decimal**. All share/USDC accounting in this
  contract goes exclusively through the `IERC20` (6-dec) interface — we never
  touch native balance and never assume 18 decimals. A "complete set" is
  parameterized in *sets*; 1 set locks `10**usdcDecimals` base units of the
  ERC-20 token (read once from the token, not hard-coded, so the contract is
  correct regardless of the deployed token's decimals).
- **Transfers can revert** on Arc to forbidden targets even with balance. We use
  the standard `IERC20.transfer/transferFrom` and let reverts propagate (no
  silent swallow); accounting is updated only after a successful pull/push.
- **EVM-compatible** — standard Solidity / `ecrecover` semantics hold; nothing
  Arc-specific is needed for this custody-independent core.

## Design

```solidity
contract CalibreMarket {
    enum Outcome { UNRESOLVED, YES, NO }
    struct Market { bool exists; Outcome outcome; }

    IERC20 public immutable usdc;
    uint256 public immutable usdcUnit;        // 10**usdc.decimals(), one set's USDC cost
    address public resolver;                  // privileged settle/create role

    mapping(uint256 => Market) public markets;                 // chainMarketId -> state
    mapping(uint256 => mapping(address => uint256)) public yesBalance;
    mapping(uint256 => mapping(address => uint256)) public noBalance;

    function createMarket(uint256 id) external onlyResolver;
    function mint(uint256 id, uint256 sets) external;          // pulls sets*usdcUnit USDC
    function transferShares(uint256 id, bool isYes, address to, uint256 amount) external;
    function resolve(uint256 id, Outcome outcome) external onlyResolver;  // one-shot, YES|NO only
    function redeem(uint256 id) external;                      // winning shares -> USDC, post-resolve
}
```

Decisions:

- **Self-contained share accounting** (no ERC-1155/ERC-20 share tokens). The
  issue asks for "per-user share accounting" and complete-set mint/redeem; full
  tokenized shares are not in scope and would add a dependency. `transferShares`
  satisfies the "share transfer" requirement minimally. W1.2's voucher flow
  credits/debits these same balances, so the storage layout is the extension hook.
- **`usdcUnit` read from `token.decimals()` at construction** rather than
  hard-coding `1e6`, so the contract is correct against any ERC-20 USDC and the
  6-dec interface is respected explicitly (W8 §3).
- **Resolver as a single mutable address** with an `onlyResolver` modifier and a
  `setResolver` guarded by the current resolver (lets W1.3 hand off from the
  deploy key to the backend signer). The issue calls for "an authorized resolver
  role"; a single role is the smallest thing that satisfies it.
- **Minimal vendored `IERC20`** instead of pulling OpenZeppelin — the scaffold
  vendors only forge-std and the issue's parsimony bar discourages new deps.
- **Redeem burns the winning-side balance and pays `winningShares * usdcUnit`.**
  Losing shares become worthless (no transfer). Solvency: each mint locked
  `sets*usdcUnit` and created `sets` YES + `sets` NO; after resolve only one side
  redeems, so total payout ≤ total locked. Invariant test asserts contract USDC
  balance ≥ outstanding winning-side obligations throughout.

## Tests (`forge test` in `contracts/`)

- `test_MintLocksUsdcAndCreditsBothSides` — mint pulls `sets*usdcUnit`, credits YES+NO.
- `test_ResolveThenRedeemWinningSide` — resolve YES, YES holder redeems 1:1, NO worthless.
- `test_RevertWhen_DoubleResolve` — second resolve reverts.
- `test_RevertWhen_RedeemBeforeResolve` — redeem on UNRESOLVED reverts.
- `test_RevertWhen_NonResolverCreates` / `_NonResolverResolves` — access control.
- `test_RevertWhen_MintUnknownMarket` — mint on a non-existent market reverts.
- `test_TransferShares` — share transfer moves balances.
- `testFuzz_SolvencyInvariant` — after arbitrary mint/transfer/resolve/redeem
  sequences, contract USDC balance ≥ sum of unredeemed winning shares * usdcUnit.

Run: `cd contracts && forge build && forge test -vvv`.

## Risks

- Decimals mismatch is the headline Arc risk; mitigated by reading
  `token.decimals()` and never touching native balance.
- Reentrancy: redeem updates balance-to-zero **before** the external
  `transfer` (checks-effects-interactions); no reentrancy guard dependency needed.
- Deployed Arc-testnet address is a W1.3 (#423) deliverable — this PR ships the
  contract + tests + deploy script; the README records a placeholder for the
  address that W1.3 fills in.
