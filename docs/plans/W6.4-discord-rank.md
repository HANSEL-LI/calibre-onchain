# W6.4 — Rank-ladder bucketer + ENS-driven Discord role-sync

Issue: HANSEL-LI/Calibre#431 (W6.4). Target repo: `calibre-onchain`
(`ranking/` Python + `discord-bot/` TypeScript). Builds on the merged W6.2 ENS
gateway (#429); needs nothing else. Subnames (#430) are degraded-around: the bot
resolves top-level `<name>.calibre.eth` only and notes the W6.3 clan-nesting
enhancement.

## What we build

Two cooperating pieces, no UI, no DB:

1. **`ranking/` (Python lib)** — the *real* rank-ladder bucketer that #419's
   interim in-module bucketer is meant to be replaced by. Pure functions:
   - The 7-tier ladder, **mechanic-locked** per overview F5:
     `Static → Hunch → Read → Edge → Sharp → Seer → Oracle`.
   - `tier_for_percentile(pct)` — maps a [0,1] skill **percentile** to a tier,
     with the top tier (`Oracle`) deliberately scarce.
   - `bucket(scores)` — given a mapping of `user_id → recency-decayed Brier
     skill score`, rank them, compute each one's percentile, and return their
     tier. This is what the private app calls so tiers are computed with the
     exact code judges can read; the resulting tier string is written to the
     `gg.calibre.rank` ENS text record.
   - The canonical `gg.calibre.*` text-record **key schema** (the scaffold
     names this lib as the schema owner; the gateway already mirrors it).

2. **`discord-bot/` (TypeScript)** — the role-sync service:
   - `/link <name>.calibre.eth` slash command: resolves the name's
     `text('gg.calibre.rank')` via a **standard ENS client (viem)** — which
     follows CCIP-read (EIP-3668) to the W6.2 gateway transparently — then maps
     the tier to a Discord role and assigns it.
   - Periodic re-sync loop (interval) re-resolves every linked member and
     reconciles roles, so a rank change flips the role on the next pass.
   - **Reads ENS only.** Zero calibre-API calls — verifiable from the code and
     network. ENS is the credential source.
   - **Privacy:** only tier/rank-derived roles. Never reads or exposes ROI,
     P&L, or open positions even though the profile exposes some of them.

## Decisions (rationale)

- **Ladder names are the canonical `rank` values.** F5 locks the mechanic and
  the 7 names; the gateway scaffold currently illustrates `tier` with Valorant-
  style placeholders ("diamond"), but the ranking lib is the schema owner, so it
  emits the ladder names and the bot's tier→role map keys on them. Unknown /
  unset rank → no role assigned (graceful, no error).
- **Percentile bucketing, top tier scarce.** Cutpoints are ascending percentile
  thresholds; `Oracle` occupies only the top ~2% so it stays a scarce signal.
  Boundaries are module constants (one source of truth, judge-readable).
- **viem `getEnsText` for resolution**, not a bespoke CCIP client: it is the
  standard third-party path and exercises the full ENSIP-10 → OffchainLookup →
  W6.2 gateway chain. RPC + ENS registry come from env (testnet placeholders).
- **Top-level names only.** #430 (subnames) is in flight; clan-nested
  `<user>.<clan>.calibre.eth` is a W6.3 enhancement. The bot resolves
  `<name>.calibre.eth` and degrades gracefully. Noted as a follow-up.
- **In-memory link registry.** "Hosting beyond the demo window" is explicitly
  out of scope; the linked-member map lives in memory for the demo. A restart
  re-links on next `/link` (or could load from a JSON file — not built).
- **discord.js is the runtime dep**; resolution is split into a pure,
  dependency-light `rank.ts`/`roles.ts` core that is unit-tested with `node
  --test` (same convention as `gateway/`), so tests run without a live Discord
  connection or RPC.
