# W6.2 — ENS offchain resolver / CCIP-read gateway

Tracker issue: `HANSEL-LI/Calibre#429` (targets this public repo).
Part of the ETHGlobal NYC 2026 Continuity umbrella (`HANSEL-LI/Calibre#415`).
Builds on the W0 scaffold's `gateway/` package; consumes calibre's Seam 2
public profile API (`HANSEL-LI/Calibre#419`, merged to calibre `master`).

## Goal

A CCIP-read (ENSIP-10 / EIP-3668) offchain resolver gateway serving
`*.calibre.eth`. A standard ENS client (viem/ethers) calling `resolve()` on the
on-chain `calibre.eth` resolver hits an `OffchainLookup`, follows it to this
gateway, and gets back a **signed** answer for:

- `addr(node)` / `addr(node, 60)` ← the profile's `wallet_address`
- `text(node, key)` for the `gg.calibre.*` / `com.discord` record keys.

The gateway is the source of truth's mouthpiece: it reads calibre's public
profile API and never touches the DB. "Updating a text record" = updating
calibre's DB — free and instant, resolvable by any ENS-aware client.

## Scope (from #429)

In:
- TS gateway implementing CCIP-read: decode the wildcard `resolve(name, data)`

## Record-key mapping (Seam 2 JSON → ENS records)

The profile API returns
`{ display_name, tier, brier_skill, roi, pnl, wallet_address, discord_handle, riot_id, clan }`.

| ENS query | Source field | Notes |
|---|---|---|
| `addr(node)` / `addr(node,60)` | `wallet_address` | coinType 60 (ETH); empty `0x` if null |
| `text(node,"gg.calibre.rank")` | `tier` | always present (e.g. `"unranked"`, `"diamond"`) |
| `text(node,"gg.calibre.brier")` | `brier_skill` | omitted (`""`) when null |
| `text(node,"gg.calibre.roi")` | `roi` | omitted when null |
| `text(node,"com.discord")` | `discord_handle` | omitted when null |
| `text(node,"gg.calibre.riot")` | `riot_id` | omitted when null |
| `text(node,"gg.calibre.clan")` | `clan` | omitted when null |

A non-existent / not-opted-in / bot subname → the profile API 404s → the
gateway answers with the **empty value** for the record (`0x` for addr, `""`
for text). This is *indistinguishable* from "record unset" and leaks nothing
(success criterion: no enumeration oracle).

## Signature scheme (ENS offchain-resolver, EIP-3668)

The gateway signs `keccak256(abi.encodePacked(0x1900, target, expires:uint64,
keccak256(request), keccak256(result)))` with `GATEWAY_SIGNER_KEY`, where
`target` is the resolver contract address, `request` is the original
`resolve(name,data)` calldata, and `result` is the ABI-encoded record answer.
Returns `abi.encode(bytes result, uint64 expires, bytes sig)` in the HTTP body
`{ "data": "0x..." }`. The reference resolver verifies the signer against an
allowlist before returning `result` to the client. This matches
`ensdomains/offchain-resolver` so any compliant ENS client follows it.

## Decisions

- **viem over ethers** for ABI encode/decode + signing: viem ships
  tree-shakeable `encodeFunctionData`/`decodeFunctionData`/`encodeAbiParameters`
  and `privateKeyToAccount(...).sign({hash})`, is the modern ENS-ecosystem
  default, and keeps the dependency surface tight. The signing math is the
  standard `ensdomains/offchain-resolver` scheme so any ENS client interoperates.
- **express for the HTTP server**: one tiny POST endpoint (EIP-3668 sender/data)
  + a `/health`. Smallest dependency that gives JSON body parsing + routing;
  matches the "stateless, a crash is a redeploy" operational note in the issue.
- **`node:test` (built-in runner) for tests**, no Jest/Vitest: zero extra deps,
  runs under Node 20+, and the gateway logic is pure functions + one fetch we
  stub. Keeps the public package legible.
- **Reference resolver `.sol` is a documented artifact, not deployed here**: the
  issue's "set the offchain resolver for `calibre.eth`" is a demo-day on-chain
  op gated on the ENS-booth-designated resolver/testnet, which doesn't exist yet
  — so the resolver *address* is a config placeholder (`GATEWAY_RESOLVER_ADDRESS`)
  and deploying is BLOCKED on the booth. The contract stub makes the signature
  scheme self-documenting and lets W6.x wire a real address later.
- **404 → empty record, not an error**: satisfies the "non-existent subname
  resolves to nothing (no leak)" success criterion and matches how ENS clients
  treat an unset record. The profile API already collapses bot/not-opted-in/
  unknown to an indistinguishable 404, so the gateway just mirrors that as empty.
- **Subname parsing = leftmost label only** for this issue: `demo.calibre.eth`
  → `display_name = "demo"`. Clan-nested `<user>.<clan>.calibre.eth` is W6.3's
  schema; out of scope here.

## Risks

- The on-chain resolver address / testnet is ENS-booth-designated and not yet
  known → kept as an env placeholder; the gateway runs and is testable without
  it (signing uses whatever `target` the resolver passes in `extraData`). Tests
  use a fixed test address. **BLOCKED (owner/booth):** deploying the resolver +
  pointing `calibre.eth` at the gateway URL.
- `CALIBRE_PUBLIC_API_BASE` defaults to the prod base in `.env.example`; the
  gateway is stateless and never authenticates (public tier), so no secret risk.
