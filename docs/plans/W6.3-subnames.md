# W6.3 — ENS subnames + clan-nested registry + ranking lib

Tracking issue: `HANSEL-LI/Calibre#430` (W6.3). Builds on the merged W6.2 ENS
offchain-resolver gateway (#429). Target repo: **calibre-onchain**. Custody-independent
(ENS leg) — lands regardless of the Saturday checkpoint.

## Goal

1. A canonical, dependency-free **ranking lib** (`ranking/`) that maps a skill
   percentile to a named tier (`tier_for_percentile`) and owns the canonical
   `gg.calibre.*` ENS text-record **key schema** every consumer reuses.
2. The gateway resolves **clan-nested subnames** `<user>.<clan>.calibre.eth`
   (per overview F4) in addition to flat `<user>.calibre.eth`, reusing the W6.2
   CCIP-read signing path and the Seam-2 profile API as source of truth — without
   introducing an enumeration oracle.

## Scope (issue #430 + the F4 clan-nesting extension named in the builder prompt)

In:
- `ranking/` (Python, pure, dependency-free):
  - `tier_for_percentile(p: float) -> str` — percentile in `[0,1]` → named tier.
    Tier ladder from overview F5: `Static → Hunch → Read → Edge → Sharp → Seer →
    Oracle` (scarce top tier). Percentile is "higher = better" (1.0 = best
    forecaster), matching the leaderboard's Brier-skill-score orientation.
  - `TEXT_KEYS` — the canonical set of `gg.calibre.*` (+ `com.discord`) record
    keys, the single source of truth the gateway's mapping must agree with.
  - Helpers: `all_tiers()` (ordered, scarcest last) so consumers (Discord W6.4)
    enumerate the ladder without re-listing it.
- `ranking/README.md` — document the tier ladder, the percentile contract, and
  the `gg.calibre.*` key schema (success-criterion: "key schema documented").
- `ranking/tests/` — unit tests (boundaries, monotonicity, clamping, key set).
- `gateway/src/resolver.ts` — extend `displayNameFor` to also accept the
  clan-nested shape `<user>.<clan>.calibre.eth` (4 labels) → the **user** leaf,
  while flat `<user>.calibre.eth` (3 labels) stays as-is. The bare clan parent
  `<clan>.calibre.eth` is structurally identical to a flat user name, so it
  resolves as a user lookup and yields empty for a non-profile clan label — no
  new endpoint, no enumeration oracle.
- `gateway/README.md` + repo `README.md` — document the nested shape + point the
  record-key table at `ranking/` as the canonical schema owner.
- `gateway/test/resolver.test.ts` — add clan-nested resolution cases.

## Decisions made

- **Tier ladder = F5 names** (`Static…Oracle`), not Valorant rank names (Riot IP
  + collision with users' in-game rank — overview F5 is explicit). 7 tiers; the
  top two (`Seer`, `Oracle`) are deliberately scarce (narrow top percentile bands).
- **`tier_for_percentile` is pure + dependency-free** — no calibre import, no
  numpy. It consumes an *already-computed* percentile so the lib carries no DB or
  scoring-pipeline coupling (the composite-axis scoring that *produces* the
  percentile stays in calibre's `leaderboard_scoring.py`; this lib only buckets).
- **Percentile orientation is higher = better** (1.0 = top forecaster), matching
  Brier-skill-score. `0.0` is the floor tier, `1.0` the top tier; out-of-range
  inputs clamp rather than raise (a resolver must never 500 on a record read).
- **Clan-nesting reuses the existing user lookup**: `<user>.<clan>.calibre.eth`
  resolves the *user* leaf (leftmost label) — the clan label is namespacing, not
  a second DB lookup. This keeps Seam 2 unchanged and preserves the
  no-enumeration property (unknown user → empty, same as today). The clan
  membership *cross-check* (does this user actually belong to this clan?) is a
  W6.4 concern once a clan endpoint exists; this weekend the nested name is an
  addressing convenience that resolves to the user.
- **No explicit claim flow code** — #430 allows an implicit claim and defers an
  explicit affordance to "if time allows"; building none keeps the change to the
  issue's core (lib + schema + resolution) and avoids frontend (which would be
  W2.2 bundle-gated in the private app anyway).
- **Gateway mapping stays the gateway's view of the schema** — `ranking/` is the
  Python source of truth for the key *set*; the TS gateway can't import a Python
  module, so the README cross-references `ranking/` as canonical and the gateway
  retains its existing TS `TEXT_RECORD_KEYS` map, which a doc note now ties to the
  Python `TEXT_KEYS`. A divergence test would need a shared schema file; that is
  out of scope (no behavior change to W6.2's mapping).

## Risks

- The Python `TEXT_KEYS` and the TS `TEXT_RECORD_KEYS` can drift. Mitigation:
  both READMEs name `ranking/keys.py` as canonical and the gateway's table now
  says "mirrors `ranking/`". A programmatic shared-schema check is deferred.
- `GATEWAY_RESOLVER_ADDRESS` / a deployed registry address is still an env
  placeholder (booth-gated, same as W6.2). No new deploy artifact needed — the
  nested resolution is pure decode logic exercised by `node:test`.
